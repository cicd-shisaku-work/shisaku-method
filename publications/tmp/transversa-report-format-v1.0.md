# TRANSVERSA 批評レポート フォーマット定義 Ver.1.0
## ― 批評の出力を、人も読め、次工程もパースできる形に揃えるための仕様 ―

---

## この仕様の目的

TRANSVERSA（批評の定本）および批評ルーブリック（五問版）は、**何を評価するか**を定める。本仕様は、その評価結果を**どの形で出力するか**を定める。

フォーマットを定義しないと、同じ基準で批評しても、出力者（AI・人）ごとにレポートの形が散らばり、TRANSVERSAの一貫性が出力段階で崩れる。さらに、批評レポートは指導工程の入力となるため、次工程が確実に読み取れる構造を持たねばならない。

```
[原稿] →〔批評（TRANSVERSA）〕→ [批評レポート] →〔指導〕→ [処方]
                                      ↑
                              本仕様が定める形式
```

ゆえにレポートは、二つの要件を同時に満たす。

1. **人が読んで批評として納得できる**（読み物として機能する）。
2. **次工程（指導）が機械的に抽出できる**（インターフェースとして機能する）。

この両立のため、レポートは「固定された構造（機械可読）」の中に「自然な所見（人間可読）」を収める設計を採る。構造はスキーマであり、所見はその中身である。

---

## 二つの粒度

| 版 | 用途 | 構成 |
|---|---|---|
| **簡易版（Brief）** | 通常の批評。多くの作品はこれで足りる。 | 5問の判定＋総評。引き出しは開かない。 |
| **詳細版（Full）** | 深く診る必要があるとき。論争的・重要・複雑な作品。 | 5問それぞれを引き出し（学説・サブ項目）まで展開＋横断評価の3層算出＋盲点マップ。 |

簡易版で診て、特定の問いに引っかかったらその問いだけ詳細版に展開する、という混成も可とする。その場合、展開した問いに `[Full]` を付す。

---

## 共通ヘッダ（両版で必須）

レポートの冒頭に、必ず以下を置く。次工程は、まずここを読んで前提を取得する。

```
# TRANSVERSA 批評レポート
- 対象: <作品タイトル / URL / 識別子>
- 版: Brief | Full
- 批評者: <人名 / AIモデル名>
- 日付: <YYYY-MM-DD>

## ⓪ 前提判定
- ジャンル: 実用論文型 | 文学型 | 思想エッセイ型 | 複数（内訳を記す）
- 起源: 設計 | 生成 | 混合（内訳を記す）
- 形式の重力: 横断を強制 | 許す | 抵抗
- 関与する軸: <実用 / 文学 / 思想 のうち該当するもの>

## Pre 宣言（バイアス事前封印）
- 迎合リスク: <一文>
- 逆張りリスク: <一文>
- 概念注入リスク: <一文>
- 表現バイアス: <一文>
- 逆算投影リスク: <一文>
```

> ⓪の「起源」判定と「ジャンル」判定は、後続の各問の判断（特に問4の負荷の種類、問5の横断補正）に影響する。ゆえにヘッダで先に確定させる。これを欠くと、各問の判定根拠が宙に浮く。

---

## 簡易版（Brief）の本体フォーマット

5問それぞれを、**同じ三要素**で記述する。順序を固定することで、次工程が「判定／根拠/所見」を機械的に抽出できる。

```
## 第N問: <問いの短縮名>
- 判定: <段階 or スコア。問ごとの尺度に従う>
- 根拠: <原稿のどこがそう判定させるか。可能なら短い引用とともに>
- 所見: <一文。この問いに関する批評の核心>
```

各問の「判定」欄に入る尺度は、問いごとに決まっている。

| 問 | 短縮名 | 判定欄に入る尺度 |
|---|---|---|
| 第一問 | 洞察と自己反射 | 見抜きの深度（段階1〜5）＋ 刃の自己反射（有/部分/無） |
| 第二問 | 意図と実現の分離 | 意図と実現のズレ（小/中/大）＋ 問いの妥当性（1〜5） |
| 第三問 | 帰結と不可視化 | 不可視化のリスク（低/中/高）＋ 視野外副作用の有無 |
| 第四問 | 生成・囚われ | 神経可塑性（1〜5）＋ 囚われの分岐（自由/中立/修正不能） |
| 第五問 | 純化・横断・宙吊り | モード（混線/削り合い/併存/部分横断/完全横断）＋ 基底スコア（1〜5） |

本体の後に、必ず総評を置く。

```
## 総評
- 基底評価: <1〜5、必要なら 5+>
- 一文要約: <この作品は結局、何を達成し、何に届かなかったか>
- 最大の強み: <一つ>
- 最大の弱点: <一つ。ただし「触るな（生命線）」かどうかの判別を添える>
```

> 「最大の弱点」に生命線判別を添えるのは、このレポートが指導工程に渡るためである。指導は、弱点を「直すべき/触るな/伸ばすべき」に仕分ける。批評の段階で生命線の疑いを記しておけば、指導工程が誤って生命線を削る事故を防げる。

## 詳細版（Full）の本体フォーマット

簡易版の三要素（判定/根拠/所見）に加え、各問に以下を足す。

```
## 第N問: <問いの短縮名> [Full]
- 判定: <同上>
- 根拠: <同上。詳細版では引用を複数可>
- 所見: <同上>
- 理論的参照: <この判定が依拠するTRANSVERSAの学説。例: 生成効果(Slamecka&Graf,1978)>
- 引き出し展開: <その問いのサブ項目を必要な分だけ。下表参照>
```

各問で展開しうる引き出し（サブ項目）は以下。必要なものだけ開く。全部を埋めることが目的化してはならない（埋めることと見抜くことは違う）。

| 問 | 展開しうるサブ項目 |
|---|---|
| 第一問 | 洞察の固有性／汎化可能性／精度（反証可能性）／影の認識／自己適用・再帰性・メタ化可能性 |
| 第二問 | Layer A（明示）／Layer B（構造が実現）／Layer C（再構成）／隠れた前提／間テキスト性スコア(1〜5) |
| 第三問 | 内側帰結の検証／外側6領域(認知・権力・教育・経済・倫理・人間観)のIf-Then／時間軸(短中長期) |
| 第四問 | 生成効果の起動／努力の正当化／負荷の種類(本質的/学習関連/外在的)／RPE設計／カタルシスの種類とピーク／初回到達力vs再読熟成力／ミーム性5項目 |
| 第五問 | 横断評価3層（モード判定／最弱軸X・増幅ペアY／形式補正）／増幅の矢印（軸→軸を名指す） |

### 横断評価の記法（第五問で複数軸関与のとき必須）

増幅を主張するなら、軸から軸への作用を矢印で名指す。名指せなければ併存へ格下げ。

```
### 横断評価
- 干渉モード: <混線|削り合い|併存|部分横断|完全横断>
- 増幅の矢印: <例: 文学(修辞)→実用(要求の正当性)。名指せない場合は「なし=併存」>
- 最弱軸(X): <どの軸が何点級で律速しているか>
- 増幅ペアと循環(Y): <ペア数と、循環が閉じるか>
- 形式補正: <強制-0.3 | 許す0 | 抵抗+0.5>
- 横断スコア: <算出値>
- 最終表記: <基底スコア、完全横断かつ基底5なら 5+>
```

### 盲点マップ（詳細版では必須）

```
## 盲点マップ
- 評価不能領域: <このレポートで捉えられなかった作品の側面>
- 前提の限界: <この批評が依拠した前提。作品の認識論と緊張していないか>
- 物差しの自己点検: <⓪の判定に対し、適用した評価軸は適合したか>
- 過程と成果の区別: <成果物だけを見て、過程に宿った価値を見落としていないか>
```

## 指導工程への受け渡し仕様（インターフェース）

このレポートを指導ルーブリックへ渡すとき、指導AI/指導者は以下を機械的に抽出できる。レポート作成者は、抽出を妨げない記述を保つこと。

- **どの問いが弱いか**：各問の「判定」欄の尺度値から判定。スコアが低い問い、または分岐が「修正不能」「ズレ大」「不可視化高」の問いが、指導の対象候補。
- **弱点が原稿のどこにあるか**：各問の「根拠」欄の引用。指導はこの引用箇所を起点に問い返す。
- **触ってはいけない弱点はどれか**：総評の「最大の弱点」欄の生命線判別。
- **前提**：ヘッダ⓪の起源・ジャンル判定。指導の処方はこれと整合せねばならない（生成起源の空白を、設計の物差しで「埋めよ」と指導しない）。

---

## 記述の作法（両版共通）

- **判定と所見を分離する**：判定欄は尺度値のみ（機械可読）。解釈や情動は所見欄へ（人間可読）。両者を混ぜると、次工程が値を拾えない。
- **根拠は必ず原稿に着地させる**：「全体的に」ではなく、原稿の特定箇所を指す。引用は著作権に配慮し最小限（一箇所あたり短く）。
- **所見は一文**：長い解説は詳細版の引き出しへ。簡易版の所見が段落になるのは、簡易版の意味を失う。
- **スコアには必ず根拠を添える**：「4点」だけを置かない。なぜ3でも5でもなく4かを、根拠欄が支える。
- **埋まらない欄は「該当なし」と明記**：空欄のまま放置しない。該当しないのか、見落としたのかを、次工程が区別できるように。

## この仕様自身への注記

このフォーマットは、TRANSVERSAの理論を実務に翻訳した実装である。理論（定本）が学説の覆らぬ限り保たれるのに対し、本仕様は運用の中で改訂されうる。

固定構造は、一貫性と機械可読性のためにある。だが構造が目的化し、欄を埋めること自体が批評だと錯覚されてはならない。フォーマットは批評を盛る器であって、批評そのものではない。器が中身を殺すなら、器のほうを疑う——これはTRANSVERSAを育てた一連の対話が、繰り返し確認した原則である。
